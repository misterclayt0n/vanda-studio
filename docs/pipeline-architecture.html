Vanda Studio · Architecture

The Autonomous Pipeline & Brand Memory

How Vanda observes, remembers, decides, creates, and publishes — built so that discernment is a property of the system, not of a prompt. The pipeline is data, the memory is the moat, and Convex is the substrate.

Scope  observe → consolidate → plan → create → publish Substrate  Convex + AI SDK + OpenRouter Status  design
01

The three principles

Everything below follows from three decisions. If you only remember these, you remember the architecture.

Principle 01

The pipeline is data, not a process

Each stage is a stateless function that reads upstream rows and writes downstream rows. The Convex scheduler drives transitions; reactivity powers the live view for free.

Principle 02

Discernment is structural

Raw signals are physically incapable of causing a post. They can only nudge beliefs. Only consolidated beliefs that clear evidence, fit and saturation gates become ideas.

Principle 03

The memory is the product

Brand understanding is the moat. It lives in your own inspectable, editable, per-account tables — never outsourced to a black-box agent.

02

We don't adopt an agent framework

The instinct to reach for eve (Vercel) or flue is understandable, but both want to own the orchestration substrate — durable execution, scheduling, state, human-in-the-loop. Convex already is that substrate. Layering either on top means two schedulers, two state stores, two notions of "a durable step" to reconcile.

The model loop you'd want from either is already solved by AI SDK v5, which is already in the repo (ai + @openrouter/ai-sdk-provider). generateText({ tools, stopWhen }) is the agent loop. So we build on what we have: AI SDK (the loop) + Convex scheduler/crons/actions (the substrate) + @convex-dev/agent (conversations).

flue is "deploy anywhere," but its harness wants a long-lived process — Convex actions are short-lived and time-boxed, so "adopt flue" really means "stand up a separate worker host beside Convex," then sync two state stores. That is more infra, not less.

03

The loop

The canonical pipeline is observe → plan → create → publish. Internally there are two more mechanisms that make it trustworthy: a consolidate step that turns raw signals into memory, and a learn step that feeds outcomes back. Consolidate is not a user-facing stage — it's the connective tissue between observing and deciding.

The "watch it live" Automático screen is free — it is just useQuery subscriptions over these tables.

04

Perception ≠ deliberation

The failure mode — "someone mentioned a dog, so post about a dog" — cannot be fixed with a better prompt. "Be discerning" is hope, not architecture. The fix is to split perception (signals updating beliefs) from deliberation (deciding what to make), with a memory layer between them that the planner reads instead of the raw stream.

The planner never sees a signal — it sees a belief and a policy. That is discernment as a property of the system.

05

The memory model

Hermes (Nous Research) is the right inspiration — curated, bounded memory plus a reflection loop that decides what's worth keeping. But Hermes uses flat markdown, and you can't threshold or decay markdown. Discernment needs exactly that. So: structured tables for the logic, vectors for retrieval, a journal for narrative continuity. We steal the pattern and own the implementation.

Brand Canon
Stable identity: voice, offerings, hard constraints, recurring characters. The golden retriever is canon, not a trend. Owner-confirmed — the fit gate.
rarely changes
Beliefs
Probabilistic claims about the world: { statement, confidence, evidenceCount, supportingSignalIds, lastReinforcedAt }. Signals reinforce or contradict these; the planner reads these.
daily · decays
Themes
Narrative threads with momentum and a history of what Vanda already posted: { name, momentum, lastPostedAt, postCount }. The saturation guard — memory of its own actions.
evolves
Outcomes
What Vanda made → what happened: { postId, themeId, metrics, learning }. Closes the loop; down-weights what flopped. Without it, Vanda never improves.
append-only
Playbook
The discernment rules, as data: cadence, evidence thresholds, safety, voice. The HOW. Owner-set and learned — editable, not hardcoded.
curated
Journal
Short reflection notes per cycle (Hermes-style narrative). Continuity between reflections; the human-readable "what Vanda is thinking."
append-only

Beliefs and themes are retrieved into the relevant calls; canon and playbook are almost always in context; a vector index over canon + past winning posts feeds create.

OWNER Memory is editable, and that is a feature. Because beliefs are rows, the owner can correct them — "we stopped winter combos" → belief retired → planning instantly stops suggesting it. That is simultaneously the inspectable-autonomy thesis and your highest-quality training signal. Surface it as a "what Vanda knows about your brand" panel in Perfil.
06

How each stage reasons

Critical calibration: only two of the stages think. Don't agent-ify everything — observe and publish are deterministic.

Observe · 0 agents

ETL, not reasoning. N deterministic source-adapters (comments, mentions, insights, competitor posts, trends) run in parallel, dedup against a dedupeKey, and write signals. Optionally one cheap salience score. Spawning an agent to fetch data is pure waste.

Consolidate · cheap model, frequent

Framed strictly as bookkeeping, never ideation. For each new signal: does it support, contradict, or introduce a belief or theme? Bump confidence (independent sources weigh more; the same user repeating is not new evidence), adjust momentum, fork on contradiction. Decay and counting are deterministic code — only the "does this support that?" judgment is a model call. Output schema is belief/theme deltas plus a journal note. It is structurally forbidden from emitting a post idea.

Plan · best model, slow cadence

Two-phase, generate-then-critique:

  • Ideate — from beliefs above threshold and rising themes, propose candidates.
  • Critique (ideally a separate, skeptical call so it isn't defending its own ideas) — score each against canon-fit, playbook (cadence/safety), saturation (theme lastPostedAt), and outcome priors. Below bar → rejected, with a reason, persisted. Above → suggestion with a full reasoning trace and provenance.

Persisting rejections is a feature: the owner sees "Vanda considered a dog post but skipped it — only 2 mentions, posted dogs 5 days ago." That is how you debug judgment.

Create · one durable workflow per post

A @convex-dev/workflow run — the one place durability earns its keep, because image generation is slow and flaky. Steps: retrieve a context bundle (vector search over canon + exemplar posts + the suggestion's signals) → generate caption + image prompts → N image-gen tool calls → compose. One agent, several tool calls — not a swarm. This is where context retrieval lives.

Publish · 0 agents

Pure Graph API. ctx.scheduler.runAt(scheduledFor, publish) fires; check content_publishing_limit (the cap is 25/50/100 per rolling 24h depending on account — read it live, never hardcode; a carousel counts as one post), build container(s) → poll status_code until FINISHED → media_publish → store the external id.

Learn · post-publish

Post metrics flow into outcomes, which feed the next consolidate pass — closing the loop so themes and formats that underperformed get down-weighted.

07

The golden retriever, walked through

Same signal, opposite outcomes — because memory decides, not the prompt.

Day 0 · one comment
"Your golden is adorable." → one signal. Consolidate bumps the belief dog content resonates 0.40 → 0.45, count 1 → 2, attaches to theme Dog (steady). No suggestion.
Plan ideates a dog post → critic kills it: 2 signals < 3, conf 0.45 < 0.6, posted dogs 5d ago. Stored as rejected. ✗ Correctly does nothing.
Over 2 weeks · accumulation
6 more dog signals across independent users, a shared dog reel, and an owner-uploaded dog photo. Belief → 0.82, count 9, theme Dog momentum rising, no recent dog post.
Next plan cycle · decision
Candidate "meet our resident golden — winter edition." Passes evidence + canon (pet-friendly, dog is canon) + no saturation + positive dog-format outcome prior.
→ suggestion, with reasoning. ✓ Good idea, right moment.

The architecture produced the judgment. The prompt never had to be clever.

08

Control = status gates, not paused processes

Control is per-item, never "approve the whole pipeline." Model it as data on the suggestions row — the load-bearing entity:

  • Assumir / Eu faço → owner = user (Vanda stops touching it).
  • Vanda faz → owner = vanda, status creating, schedule create.
  • Dispensar → status dismissed.
  • Sensitive actions (reply to a negative comment, anything risky) → created with requiresApproval = true, status needs_you.

The scheduler simply never fires publish or reply for a row in needs_you. No blocked process to babysit, no orphaned in-flight state. Flip the field, the work proceeds. Crash-safe by construction — this is how we beat eve's "pause the workflow for approval."

10

Built to expand (video, reels, tweets, stores)

Keep the pipeline core generic and push platform/format specifics behind two interfaces. Plan stays format-agnostic. Adding a format is adding a strategy — never touching the core.

observe

SourceAdapter

fetch(account) → Signal[]. IG comments/mentions/trends/competitors today; X-mentions, store-orders later. New source = new adapter.

create

CreatorStrategy

Keyed by post.type. Feed/image (carousel) now; reel/video later — images generalizes to a media unit that holds a clip. New format = new strategy.

publish

Publisher

Keyed by platform. instagram now; x, etc. later. Schema is already platform-tagged. New platform = new publisher.

Composability is the core: images are atomic, posts compose them, and a unit can be generated, uploaded or pulled from the gallery — interchangeably. That relation (images ↔ posts.imageIds) is what the whole product expands from. Effect fits cleanly: write adapters/strategies/publishers as Effect services with typed errors and retry schedules, invoked from inside the Convex action.

11

Build order

Each step is one jj commit; cargo check / tsc clean before describing.

  1. Schema + memory tables — accounts, brandCanon, seed Café Lumiar canon (pet-friendly, the golden, "combo de inverno", competitor Grão & Co).
  2. Publish end-to-end — deterministic, validates Graph API + scheduler + calendar. Ship the highest-certainty piece first.
  3. Observe — cron + IG comments/mentions adapters → signals → light up the Automático "Observando agora" feed via reactivity.
  4. Consolidate — the lynchpin and the cheapest to validate. Signals → belief/theme deltas + journal.
  5. Plan — generate-then-critique → suggestions with reasoning → the "O plano da Vanda" cards + per-item control.
  6. Create — workflow + RAG context bundle → images / posts → Galeria.
  7. Assistente + Editar — @convex-dev/agent threads.
12

Flags before building

SECRET A live, long-lived Instagram access token is committed at ./INSTAGRAM (with an account id). That is a real leaked secret — rotate it and move it to env/secret storage. The schema already encrypts tokens at rest in instagramConnections; this plaintext file is the leak, not the design.
TRENDS "Tendências" has no first-party source. The observe step needs a concrete adapter (hashtag search via Graph, or a third-party trends API). Decide this before promising the signal in the UI — it's the one observe source without an obvious API.
QUOTA The publishing cap is account-specific (25 / 50 / 100 per rolling 24h). Always read content_publishing_limit live and back off; never hardcode a number.
